## Suppressing False Positives

Add an inline ignore comment on the flagged line:

```js
// codesec-ignore: SECRETS_GENERIC_API_KEY
const STRIPE_PK = 'pk_live_abc123';  // publishable key, safe to expose
```

```python
# codesec-ignore: BACKEND_PYTHON_EVAL
eval(trusted_expression)  # only called with validated AST
```

```html
<!-- codesec-ignore: WEB_INNERHTML -->
<div id="trusted-content"></div>
```

## Methodology & Limitations

This report was generated by:

1. **`codesec-scan` binary** — regex-based static analysis across {N_FILES} files ({N_LINES} lines of code). Fast, deterministic, runs in CI.
2. **Claude LLM analysis** — semantic review of findings + manual checklist covering auth logic, IDOR, SSRF, and business-logic flaws that regex cannot detect.

**What this scan covers:**
- Secrets and credential exposure
- Injection vulnerabilities (SQLi, CMDi, XSS)
- Insecure deserialization and unsafe parsing
- Authentication and authorization weaknesses
- Cryptographic issues
- Dependency vulnerabilities (CVEs)
- Infrastructure misconfigurations

**Limitations — what this scan does NOT guarantee:**
- This is a first-pass review, not a penetration test. It does not execute the code or probe a running application.
- Regex rules have false negative rates. Novel or obfuscated vulnerability patterns may be missed.
- Business logic flaws (e.g., price manipulation, workflow bypass) require domain knowledge and manual testing.
- Race conditions and timing attacks require dynamic testing to confirm exploitability.
- Third-party services, APIs, and external integrations are out of scope unless their configuration files are present.

For high-assurance systems, supplement this review with dynamic testing (DAST), manual penetration testing, and threat modeling.
